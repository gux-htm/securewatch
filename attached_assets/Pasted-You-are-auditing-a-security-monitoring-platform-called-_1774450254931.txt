You are auditing a security monitoring platform called SecureWatch. Below is a complete inventory of features that are ALREADY IMPLEMENTED. Your job is to check if the system you are working on has these features. For any that are missing, implement them.

---

## AUTHENTICATION
- 2-step login: Step 1 = credentials → short-lived pre-auth token (5 min TTL). Step 2 = TOTP code + device fingerprint → full JWT
- Rate limiting: max 10 login attempts per IP per 15 minutes (in-process cache, no Redis)
- MFA lockout: 3 failed TOTP attempts locks the account
- Generic "Access Denied" response on ALL auth failures — never reveal the reason
- Three-layer verification before JWT is issued:
  - Layer 1: Account must be ACTIVE
  - Layer 2: Source IP must match a registered network zone
  - Layer 3: Device must be registered and not BLACKLISTED

## ALERT INBOX
- Unified alert list with severity levels: CRITICAL, HIGH, MEDIUM, LOW, INFO
- Filter alerts by severity
- Acknowledge individual alerts (PATCH endpoint)
- Unacknowledged count badge in the navigation
- Critical alert count shown in topbar

## LIVE SESSION MONITOR
- List all active (non-terminated) sessions
- Auto-refresh every 10 seconds
- Search/filter by username, IP address, device
- Terminate session action (sets is_terminated flag, writes audit event)
- Deep-link from session row to audit log filtered by that account

---

## RESOURCE REGISTRY
- Register resources (files, directories, databases, APIs, etc.)
- Edit and archive resources
- ACL management per resource: grant and revoke access entries with permitted_actions (JSON array)
- Access event history per resource
- Hash-based integrity: baseline hash + current hash stored, comparison shown in UI

---

## AUDIT LOG (Object-Centric)
- One row per monitored object (FILE or ACCOUNT)
- Each object has a current flag: GREEN / SUSPICIOUS / CRITICAL
- History panel: event timeline per object with 5s auto-refresh
- Detail panel per event: actor info (Windows user, IP, MAC, hostname), hash before/after, hash_changed flag, HMAC-SHA256 signature, digital signature validity
- Acknowledge individual audit events
- Pagination and filtering by object type, flag, and event type
- Deep-linkable via query params (object ID in URL)
- Audit events are append-only and HMAC-SHA256 signed — no DELETE ever issued on this table

## KAFKA EVENT PIPELINE
- Integration layer (port 3001): single authenticated entry point for all incoming events
  - POST /api/v1/events — single event ingestion (source_id, source_type, tenant_id, payload)
  - POST /api/v1/events/batch — batch ingestion up to 1000 events, returns accepted/rejected counts
  - Normalises events and publishes to Kafka — no direct calls to other services
- Auth engine: Kafka consumer on topic sw.events.session — runs three-layer verification, publishes DENIED verdicts to sw.alerts.outbound
- Resource engine: Kafka consumer on topic sw.events.resource — ACL resolution (direct → group → inherited), DENY beats ALLOW, publishes violations to sw.alerts.outbound

## GO AGENT (Windows)
- Monitors a configurable directory for file system events (create/modify/delete/rename/view)
- Syncs Windows users to REST API on startup via POST /api/v1/agent/sync-users
- Fetches approved users list every 5 minutes via GET /api/v1/agent/approved-users
- Publishes HMAC-SHA256 signed file events to Kafka topic securewatch.file-events

## DATABASE SCHEMA (MySQL 8.0, mysql2/promise only)
Tables that must exist:
- tenants — multi-tenant support
- accounts — with mfa_secret, mfa_enabled, status (ACTIVE/SUSPENDED/REVOKED), failed_login_count
- devices — with fingerprint, status (TRUSTED/UNTRUSTED/PENDING/BLACKLISTED)
- network_zones — CIDR-based zones per tenant
- groups + group_members — privilege groups
- resources — resource registry with baseline_hash, current_hash
- acl_entries — permitted_actions (JSON), status (ACTIVE/REVOKED)
- integration_registry — type, method, version, status, last_seen
- active_sessions — with risk_verdict (CLEAN/SUSPICIOUS/CRITICAL), is_terminated flag
- alerts — severity, detail JSON, acknowledged flag
- audit_events — HMAC-SHA256 signed, append-only, with object_id, hash_before, hash_after, hash_changed, actor_windows_user, actor_mac, actor_hostname, digital_sig_valid, acknowledged
- audit_objects — one row per monitored object (FILE/ACCOUNT), current_flag (GREEN/SUSPICIOUS/CRITICAL), last_event_at
- file_events / monitored_file_events — file system event log
- settings — per-tenant settings (alert_threshold, session_timeout, mfa_enforced, webhook_url, smtp_*)
